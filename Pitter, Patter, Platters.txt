📝 Challenge Description

    "Suspicious" is written all over this disk image.

    Hints:

        It may help to analyze this image in multiple ways: as a blob, and as an actual mounted disk.  

        Have you heard of slack space? There is a certain set of tools that now come with Ubuntu that I'd recommend for examining that disk space phenomenon...  

🔍 Initial Triage & Enumeration

We are provided with a raw disk image partition file named suspicious.dd.sda1.
1. Mounting the Image

Our first step is to mount the disk image locally to examine its visible file system contents:
Bash

mkdir mnt
sudo mount -o loop suspicious.dd.sda1 mnt
ls -la mnt

Inside the mounted directory, we locate a file named suspicious-file.txt. Inspecting its contents reveals a clue:
Bash

cat mnt/suspicious-file.txt
# Output: Nothing to see here! But you may want to look here -->

2. Exploring File Metadata

Using fls from The Sleuth Kit (TSK), we can view the directory entries of the raw image without relying on the OS mount point layout:
Bash

fls -r suspicious.dd.sda1

The output shows a standard Tiny Core Linux setup (tce directories, mydata.tgz, etc.), but searching for the standard flag format (picoCTF{) globally across the raw image or the mounted files using standard grep returns absolutely no results.
💡 Vulnerability / Concept Analysis

The core of this challenge relies on understanding Slack Space.

Operating systems allocate disk space in fixed-size blocks (clusters/sectors). When a small file like suspicious-file.txt (only 55 bytes) is written, the OS still allocates a full sector (typically 4096 bytes). The remaining bytes in that sector are unused by the file but reserved. This hidden boundary is called slack space.

Because standard file-reading utilities stop executing exactly at the file's designated End-of-File (EOF) marker, data hidden in slack space remains invisible to standard user commands (cat, grep, etc.).

Furthermore, string matching failed because the hidden data was obfuscated using Null-byte padding (0x00 between characters) and written in reverse order (a play on the challenge title "Pitter, Patter").
🛠️ Exploitation & Resolution
1. Finding the Target Offset

Since the clue explicitly pointed to suspicious-file.txt, we can search for its content string inside the raw byte layout of the image and return its exact hexadecimal offset location:
Bash

strings -a -t x suspicious.dd.sda1 | grep "Nothing to see here!"

Output:
Plaintext

200400 Nothing to see here! But you may want to look here -->

2. Dumping the Raw Sector

Now that we have the exact location (0x200400), we bypass file system restrictions entirely and look directly at the physical storage block using xxd:
Bash

xxd -s 0x200400 -l 250 suspicious.dd.sda1

Looking right past the text string into the slack space, we see characters separated by null bytes (00). Reading the characters backwards from the bottom-right (p.o.c.i.p) upwards to the top-left yields the flag.

Note: The hex character 3c in line 00200440 is a less-than sign (<), which when reversed into correct string alignment flips into a greater-than sign (>).
3. Automated Flag Extraction

We can script a quick Python one-liner to slice the precise region out of the raw disk file, strip out the non-printable null bytes, and reverse the string cleanly:
Bash

python3 -c 'with open("suspicious.dd.sda1", "rb") as f: f.seek(0x200437); data = f.read(60); print("".join([chr(b) for b in data if 32 <= b <= 126])[::-1])'
